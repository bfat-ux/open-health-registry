Methodology

How we build registries that get used.

Three interlocking ideas: a trust model that starts honest and earns confidence progressively, an operating model that ends with transfer, and a technology stance that avoids lock-in.

The Validation Ladder

Registry data is only as useful as it is trustworthy. Most health registries treat all records as equally valid — which means they're equally suspect. OHR's validation ladder assigns an explicit trust level to every record based on the evidence that supports it.

Records start at L0 the moment they're ingested from any data source. They earn higher levels through progressive verification — each step requires harder-to-fake evidence. Regulators and system integrators can query by level, filter by level, and set minimum-level requirements for specific use cases.

L0
Mapped Record ingested from a source dataset. Existence is asserted, not verified. Suitable for planning; not for credentialing.
L1
Contact Confirmed A phone call, SMS, or in-app reply has been received and logged. The facility or provider is reachable at the listed contact.
L2
Evidence Documented GPS coordinates and a timestamped photograph from a field visit. The physical location matches the record.
L3
Regulator / Partner Verified Record cross-referenced with an official dataset from the regulatory authority or a trusted partner. Licence number or registration confirmed.
L4
High-Assurance In-person audit or biometric check-in. Reserved for high-stakes use cases such as controlled substance dispensing or prescription routing.

Why graduated trust matters

A system that treats unverified and verified records identically can't be used safely for anything that matters. By making trust level explicit and queryable, we allow regulators to enforce minimum standards (e.g. "only L3+ providers can receive e-prescriptions") without requiring that every record be fully verified before the registry becomes useful.

This also creates a path for progressive improvement: a registry that starts at L0 for 6,000 facilities is still useful for planning. As field teams verify records, the useful population for credentialing grows. The ladder makes that progress legible and auditable.

Build-Operate-Transfer

BOT is not new. It's a proven model for infrastructure development. OHR applies it to health data: we build what's needed, operate it through the transition period, and hand it off. The national authority ends up owning a running system, not a dependency.

Build

Every design decision is made with handover in mind. We use open standards, document as we go, and avoid architectural choices that would make future maintenance dependent on OHR. The codebase belongs to the partner from day one.

Operate

During the operate phase we run the system alongside counterpart staff, not instead of them. Training is built into the operating rhythm. We refine workflows with the regulator, validate data quality, and monitor for issues — while building institutional capacity in parallel.

Transfer

Transfer is a planned event, not a handoff that happens when funding ends. We define transfer criteria at the start of each engagement: what capabilities must exist, what data must meet what quality thresholds, what staff must be trained. Transfer happens when those criteria are met.

Open standards first

Every technology choice is evaluated against one question: will this make the system harder to transfer, maintain, or integrate? Proprietary databases, vendor-specific APIs, and closed formats all fail that test. OHR builds on standards that are well-documented, widely implemented, and owned by no one.

FHIR R4

Interoperability mappings for provider and facility resources. Ensures registry data can be consumed by any FHIR-compatible system.

OpenAPI 3.1

All REST surfaces documented in machine-readable specs. Integrators can generate client libraries without waiting for OHR.

PostgreSQL + PostGIS

Open-source relational database with geospatial extensions. No licence cost, no vendor lock-in, runs anywhere.

OpenStreetMap

Geospatial basemap and administrative boundary data. Community-maintained; no API dependency for core functionality.

What "no lock-in" means in practice

When OHR transfers a system to a national authority, that authority can host it on any cloud provider, modify it with any developer who knows PostgreSQL and Python, and integrate it with any system that speaks FHIR or REST. There is no OHR SDK required, no OHR licence to renew, no OHR API that goes away if we close.
