Methodology

How we build registries that national systems can actually own and trust.

Three interlocking ideas guide everything we do: a graduated trust model that starts honest and earns confidence progressively, a Build–Operate–Transfer operating model that ends with full handover, and a technology stance that eliminates vendor lock-in from day one.

The Validation Ladder

Registry data is only useful if it is trustworthy. Most health registries treat every record as equally valid — which means they are equally suspect. OHR's Validation Ladder assigns an explicit, queryable trust level to every record based on the strength of supporting evidence.

Records begin at L0 the moment they are ingested. They rise through progressive verification — each level requires harder-to-fake evidence. Regulators and integrators can query, filter, and set minimum-level requirements for different use cases (e.g., "only L3+ providers may receive e-prescriptions").

L0
Mapped Record ingested from any source dataset. Existence asserted, not verified. Useful for planning and gap analysis.
L1
Contact Confirmed Phone, SMS, or in-app reply logged. The provider or facility is reachable.
L2
Evidence Documented GPS coordinates + timestamped field photograph. Physical location matches the record.
L3
Regulator / Partner Verified Cross-referenced with official regulatory data or a trusted partner dataset. Licence or registration confirmed.
L4
High-Assurance In-person audit or biometric check-in. Reserved for highest-stakes uses: controlled substances, prescription routing, and similar.

Why graduated trust matters

A system that cannot distinguish verified from unverified records cannot be used safely for anything important. The ladder lets regulators enforce real standards immediately while still making the registry useful from day one. It also makes progress visible and auditable as verification work advances.

This approach directly addresses a recognised gap across African digital health efforts: the need for reliable, cross-organisational identity and verification layers that existing systems often lack.

Build–Operate–Transfer

We apply a proven infrastructure model to health data registries. We build what is needed, operate it with local teams during the transition, and transfer full ownership — code, data, documentation, and institutional knowledge — so the national authority ends up with a running system, not a dependency.

Build

Every design decision is made with eventual handover in mind. We use only open standards, document continuously, and avoid any architecture that would require OHR's ongoing involvement. The codebase belongs to the partner country from day one.

Operate

We run the system with counterpart staff, not instead of them. Training is embedded in daily operations. We refine workflows together, validate data quality, and build local capacity in parallel.

Transfer

Transfer is a planned milestone, not something that happens when funding ends. We define clear criteria at the start of each engagement — technical capabilities, data quality thresholds, trained personnel. Transfer occurs only when those criteria are met.

Open standards, no lock-in

Every technology choice is tested against one question: "Will this make the system harder for the national authority to maintain, modify, or integrate after we leave?" We build exclusively on standards that are well-documented, widely implemented, and owned by no one.

FHIR R4

Interoperability for provider, facility, and patient resources. Registry data works with any FHIR-compatible system.

OpenAPI 3.1

All APIs documented in machine-readable specs so integrators can generate clients immediately.

PostgreSQL + PostGIS

Open-source, battle-tested database with geospatial support. Runs anywhere with no licence fees.

OpenStreetMap

Community-maintained geospatial basemap and boundaries. No paid API dependency.

What "no lock-in" means in practice

When transfer happens, the national authority can host the system on any cloud provider, maintain it with any developer familiar with PostgreSQL and Python, and integrate it with any system that speaks FHIR or REST. There is no OHR-specific code, licence, or API that disappears when we leave.

This stance aligns with the OpenHIE community's emphasis on sustainable, locally owned health information exchange — while focusing specifically on delivering the critical registry layer with a clear exit path.
